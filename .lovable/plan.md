
문제는 “회색 설명은 맞게 고쳤는데 아래 수정안만 엉뚱하게 바뀌는” 흐름이 아직 코드상으로 그대로 남아 있어서임. 지금은 모델이 올바르게 설명해도, 마지막에 `[수정안]`만 별도 후처리되면서 다시 망가질 수 있음.

## 지금 왜 저 화면처럼 되는지

### 1. 설명 본문과 수정안은 서로 다른 경로로 처리됨
- `reply`(회색 설명)와 `suggestionNotes`(적용 버튼 아래 수정안)는 같은 답처럼 보이지만 실제로는 별도 처리됨.
- 서버가 `[수정안]...[/수정안]`만 따로 뽑아서 다시 태그 감지 + 고정패턴 후처리를 거침.
- 그래서 **설명은 맞고 수정안만 틀리는 모순**이 발생 가능.

### 2. 타겟 태그가 너무 넓게 잡힘
현재 3번 노트 같은 케이스는 실제로는:
- 계속적 용법 관계대명사
- 목적격 관계대명사
- 선행사 = `plant products`
같은 세부 프레임이 중요한데,

현재 `chatDetectUiTagFromContent()`는 이런 세부형보다 먼저 `관계대명사`로 뭉뚱그려 반환할 가능성이 큼.  
실제 로그도 `Active tags: [관계대명사], Matched 2/54 patterns`로 찍힘.

즉, 시스템은 “이건 그냥 관계대명사 패턴이네”라고 보고,
DB의 최신 `관계대명사` 패턴 하나를 강제로 적용해버림.

### 3. `pinnedByTag`가 태그당 패턴 1개만 저장함
`syntax_patterns`를 보면 `관계대명사` 태그 아래에:
- `what` 주어절 패턴
- 목적격 생략 패턴
- 주격 that 패턴
등 서로 전혀 다른 구조가 섞여 있음.

그런데 현재는 `Map<string, string>`이라서
- `관계대명사` → **가장 최근 1개 패턴만 저장**
이 구조임.

그래서 지금 케이스처럼 `which` 계속적 용법을 고쳐야 할 때도,
최신 `what` 패턴이 선택되면
후처리가 아예 그 틀로 문장을 다시 덮어써버림.

스크린샷의
`선행사를 포함한 관계대명사 which이 이끄는 명사절...`
같은 이상한 수정안은 바로 이 현상과 일치함.
즉, **AI가 그 문장을 새로 생각해서 쓴 게 아니라, 잘못 고른 고정패턴 틀에 영어만 바꿔 꽂은 결과**로 보임.

### 4. 재분석 모드에서는 가드가 약해짐
사용자가 `아니`, `틀렸어`, `다시` 같은 말을 하면 `allowReanalysis = true`가 됨.
문제는 이때:
- 이전 답 반복 방지는 일부 했지만
- “기존 프레임 유지” 가드는 꺼짐

즉, 사용자가 오류를 잘 짚어줘도
후처리가 수정안을 다시 다른 문법 틀로 비틀어도 막지 못함.

그래서 지금처럼:
- 회색 설명: 올바르게 오류 인식
- 수정안: 다른 문법 틀로 붕괴
가 동시에 생김.

## 왜 최근 수정으로도 안 잡혔는지

이전 수정들은 주로:
- 히스토리 앵커링 완화
- `___` 없는 패턴도 강제
- 태그 누락 보완
에 초점이 있었음.

그런데 이번 오류의 핵심은 그보다 더 아래 단계인:
1. **태그를 너무 넓게 분류하는 문제**
2. **태그당 패턴을 1개만 쓰는 문제**
3. **수정안 후처리가 설명보다 더 강하게 덮어쓰는 문제**
4. **재분석 시 안전장치가 풀리는 문제**
라서, 앞 수정만으로는 해결되지 않았음.

## 수정 방향

### 1. 관계사 태그를 “세부형 우선”으로 다시 분류
`관계대명사`를 바로 반환하지 말고 먼저 구분:
- 계속적용법 관계대명사
- 전치사+관계대명사
- 선행사 포함 관계대명사(what)
- 일반 관계대명사
- 관계부사
이 순서로 세분화해 exact match 우선 적용.

### 2. `pinnedByTag`를 “태그당 1개”가 아니라 “후보 여러 개”로 변경
특히 chat에서는:
- broad tag 1개 고정이 아니라
- exact subtype
- 현재 note 내용과의 유사도
- `example_sentence`와의 구조 유사도
로 랭킹해서 가장 가까운 패턴만 선택해야 함.

즉:
```text
관계대명사
 ├─ what 명사절
 ├─ 목적격 관계대명사
 ├─ 계속적 용법
 └─ 주격 that 형용사절
```
이걸 하나로 뭉개면 안 됨.

### 3. 후처리를 “강제 치환”에서 “검증 후 적용”으로 바꾸기
지금은 잘못 고른 패턴이면 그 틀을 그대로 덮어씀.

바꿔야 할 방향:
- 패턴 적용 전후를 비교
- 아래 같은 변화가 생기면 패턴 적용 거부:
  - 형용사절 → 명사절
  - 선행사 전체문장 → 명사구
  - 계속적 용법 → 일반 관계대명사
  - `what`류 선행사 포함 구조가 갑자기 생김
- 거부되면 AI 원안(raw suggestion)을 유지

즉, **고정패턴은 “형식 보정기”여야지 “의미 재작성기”가 되면 안 됨**.

### 4. 재분석 모드에서도 최소 안전장치 유지
재분석이라고 해서 모든 프레임 변경을 허용하면 안 됨.

구분 필요:
- 사용자가 “이건 관계대명사 아니고 접속사야”처럼 **분류 자체를 바꾸라고 한 경우**
- 사용자가 “선행사가 앞문장 전체가 아니라 plant products야”처럼 **같은 문법 틀 안에서 오류만 고친 경우**

후자의 경우에는 재분석이어도:
- 문법 대분류는 유지
- 세부 수정만 허용
- 패턴 후처리가 다른 문법 틀로 바꾸면 차단
으로 가야 함.

### 5. 설명 본문과 수정안의 불일치 검증 추가
회색 설명이:
- `plant products가 선행사`
- `목적격 관계대명사`
라고 말했는데

수정안이:
- `선행사를 포함한 관계대명사`
- `명사절`
이면 바로 모순임.

따라서 `[수정안]` 추출 후:
- 본문 핵심 진술과 수정안이 충돌하면
  - 수정안만 재생성하거나
  - 수정안 표시 자체를 막고
  - 원노트 유지
로 fail-closed 처리해야 함.

## 기대 결과

수정 후에는:
- 회색 설명이 맞으면 수정안도 같은 방향으로 맞아야 함
- 잘못된 generic `관계대명사` 패턴이 `what 명사절` 틀로 덮어쓰는 현상 차단
- 사용자가 오류를 정확히 짚으면, 수정안이 그 오류를 반영해서 바뀌고 다시 엉뚱한 구조로 붕괴하지 않음
- 확신이 없을 때는 틀린 수정안을 내기보다 원노트를 유지하는 쪽으로 안전하게 동작
