
구문분석 자동생성에서 문장과 무관한 문법이 나오는 근본 원인 진단 및 구조 개선 계획

1. 현재 원인 진단
- 이 문제는 모델 성능 문제가 아니라 “고정 패턴 주입 구조” 문제임.
- `supabase/functions/grammar/index.ts`에서 자동 생성 시마다 전역 고정 패턴(`syntax_patterns`, `is_global = true`)을 모두 불러와 프롬프트에 주입함.
- 그런데 관련성 판정이 너무 느슨함:
  - `extractEnglishKeywords()`가 고빈도 영어 단어를 거의 제거하지 못함
  - `relevantPatterns` 선정이 `keywords.some(...)`라서 키워드 1개만 겹쳐도 통과함
- 더 큰 문제는 `byTag`에 실제 적용 가능한 패턴만 따로 거른 뒤에도, 프롬프트에는 그보다 넓은 `relevantPatterns` 전체를 그대로 넣고 있다는 점임.
  - 즉 “실제로 적용하지 않을 패턴”도 모델은 이미 시스템 프롬프트에서 읽고 따라하게 됨.
- DB에 저장된 고정 패턴 중 다수가 “재사용 가능한 템플릿”이 아니라 “특정 문장용 해설 예시” 형태임.
  - 예: `rather than: ~라기보다는 / 'A rather than B' 구조...`
  - 이런 예시형 패턴이 전역 공유되면 다른 문장에도 오염처럼 퍼짐.
- 태그 체계도 정규화되어 있지 않음.
  - `기타`, `분사`, `강조구문`, `전치사+관계대명사` 등 자유 태그가 섞여 있어 서버가 정확히 어떤 상황에만 써야 하는지 판단하기 어려움.

2. 왜 “갑자기” 심해졌는가
- 최근 마이그레이션에서 관리자 패턴을 전역 공개로 바꾸는 흐름이 들어감.
  - `20260322093000...sql`: 전역 읽기 정책 추가
  - `20260322100500...sql`: 기존 관리자 패턴을 전역 범위로 복원
- 즉 예전엔 일부 계정/일부 상황에서만 영향이 있던 패턴이, 지금은 모든 로그인 사용자 자동생성에 공통 주입되는 구조가 됨.
- 그래서 특정 패턴 하나가 잘못 저장되면 전 사용자 자동생성 품질을 한 번에 망가뜨릴 수 있는 상태임.
- “rather than”가 반복되는 이유도 같은 맥락임:
  - 해당 패턴이 실제 DB에 전역 저장돼 있음
  - 관련성 필터가 약해서 쉽게 프롬프트에 포함됨
  - 모델이 그걸 “현재 문장에 유효한 참고 규칙”로 오인함

3. 결론
- 지금 문제의 본질은 “문장 분석”보다 “전역 패턴 주입 레이어”가 더 강하게 작동하고 있다는 점임.
- 즉 모델이 엉뚱해진 게 아니라, 서버가 자동생성 직전에 잘못된 문법 예시를 참고자료로 강하게 먹이고 있음.
- 그래서 단순 stopword 추가만으로는 불충분하고, 패턴 시스템 자체를 재설계해야 함.

4. 구현 계획
- 1단계: 패턴 타입 분리
  - 고정 패턴을 `재사용 템플릿`과 `문장 예시`로 분리
  - 자동생성에는 재사용 템플릿만 허용
  - 특정 문장 예시는 관리자 화면에서 참고용으로만 보관
- 2단계: 관련성 판정 로직 교체
  - `some()` 기반 단어 1개 일치 방식 제거
  - 구문 단위 일치(`rather than`, `not only A but also B` 등), 태그 기반 일치, 최소 매칭 비율을 함께 사용
  - 패턴에 `match_phrase` 또는 `match_terms` 같은 명시 필드 도입
- 3단계: 프롬프트 주입 경로 정리
  - `promptBlock`은 “최종 승인된 패턴”만 넣도록 변경
  - 지금처럼 `relevantPatterns` 전체를 주입하지 않게 수정
- 4단계: 태그 표준화
  - 자유 입력 태그를 줄이고, 자동생성에서 쓰는 태그는 표준 목록으로 정규화
  - `기타`는 자동생성 우선순위에서 제외하거나 매우 제한적으로만 사용
- 5단계: 관리자 패턴 정리 도구
  - 기존 전역 패턴 목록에서 문제 패턴을 점검
  - sentence-specific 패턴, placeholder 없는 패턴, generic tag 패턴을 식별
  - 자동생성 반영 여부를 관리자 UI에서 명확히 구분
- 6단계: 디버깅 가시성 추가
  - 자동생성 로그에 “이번 문장에 어떤 패턴이 주입됐는지” 남기기
  - 다시 이런 문제가 생겨도 즉시 원인 추적 가능하게 개선

5. 수정 대상
- 서버 로직: `supabase/functions/grammar/index.ts`
- 관리자 UI: `src/components/PinnedPatternsManager.tsx`
- 고정 버튼 흐름: `src/components/SyntaxNotesSection.tsx`
- 필요 시 DB 스키마: `syntax_patterns`에 패턴 타입/매칭 필드 추가

6. 기술적으로 핵심 수정 포인트
- `fetchPinnedPatterns()`에서
  - 패턴 검색
  - 관련성 판정
  - 프롬프트 주입 대상
  이 3가지를 분리해야 함
- 현재는 “대충 관련 있어 보이는 패턴 목록”과 “실제로 써도 되는 패턴 목록”이 다름에도 둘을 섞어 쓰고 있음
- 이 결합을 끊는 것이 근본 해결임

7. 기대 결과
- 자동생성이 다시 “현재 문장 자체”를 중심으로 작동
- 전역 패턴은 스타일 보정에만 쓰이고, 문장을 덮어쓰는 역할은 하지 않게 됨
- `rather than` 같은 특정 표현이 무관한 문장에 반복 등장하는 현상 제거
- 앞으로 관리자 패턴을 추가해도 전체 서비스 품질이 갑자기 무너지는 구조를 방지

8. 승인 후 진행 순서
- 먼저 전역 패턴 주입 로직을 구조적으로 분리
- 다음으로 기존 패턴 데이터 정리 기준 적용
- 마지막으로 자동생성 로그를 넣고 실제 문장 여러 개로 회귀 점검
